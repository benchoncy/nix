# AGENTS.md

## Repo Purpose

This repository is a Nix flake for provisioning the author's personal and work machines.
It targets both NixOS and macOS via `nix-darwin`, and supports both embedded Home Manager in system builds and standalone `homeConfigurations` for dotfiles-only runs.

## Important Entry Points

- `flake.nix`: defines flake inputs and the available host outputs.
- `hosts/nixos-personal/configuration.nix`: NixOS host composition root.
- `hosts/darwin-work/configuration.nix`: macOS host composition root.
- `modules/shared/default.nix`: shared base imported by both platforms.
- `modules/home/default.nix`: shared Home Manager root for the configured user.
- `modules/home/profiles/base.nix`: standalone-safe Home Manager capability composition.
- `Makefile`: convenience commands for rebuilds, flake checks, and input updates.
- `README.md`: bootstrap notes, especially for first-time Darwin setup and NixOS DisplayLink setup.
- `examples/work-overlay-mvp/`: generic recovery skeleton for a local private work overlay repo.

## Current Hosts

Hosts are declared explicitly in `flake.nix`.

- `darwinConfigurations.bstuart-mbp-m1pro`
  - system: `aarch64-darwin`
  - host module: `hosts/darwin-work/configuration.nix`
  - special args: `inputs`, `username = "bstuart"`
- `nixosConfigurations.nixos-bstuart`
  - system: `x86_64-linux`
  - host module: `hosts/nixos-personal/configuration.nix`
  - special args: `inputs`, `hostname`, `username = "ben"`, `displayname`

When adding a new machine, follow the same pattern: add a host directory under `hosts/`, keep its `configuration.nix` thin, and register the new output in `flake.nix`.

## Layout

### `hosts/`

Per-machine entrypoints live here.
Keep host files focused on selecting modules and enabling host-specific options.

- `hosts/nixos-personal/hardware-configuration.nix` is generated by NixOS and should only be edited intentionally.
- `hosts/nixos-personal/settings.nix` contains machine-local settings such as DisplayLink video drivers and keyboard layout.

### `modules/shared/`

Cross-platform system configuration.

- `default.nix` aggregates shared modules.
- `packages.nix` enables shared system integrations like 1Password.
- `options.nix` defines shared custom options.
- `home-manager.nix` wires Home Manager into the system build for `users."${username}"`.

Notable behavior:

- unfree packages are allowed globally
- the NUR overlay is enabled globally
- flakes are enabled via `nix.settings.experimental-features`
- Home Manager uses `backupFileExtension = "bkp"`

### `modules/nixos/`

Linux-only system configuration.

- `default.nix` imports Home Manager's NixOS module, the Catppuccin NixOS module, package config, and display modules.
- `display/` contains feature modules like `gnome.nix` and `hyprland.nix`.

### `modules/darwin/`

macOS-only system configuration.

- `default.nix` imports Home Manager's Darwin module and Darwin package config.
- `packages.nix` manages GUI apps with Homebrew casks.

Important: `modules/darwin/default.nix` sets `nix.enable = false` because Nix is expected to be installed and managed by Determinate Systems.

### `modules/home/`

Shared Home Manager config imported through the system modules.

- `default.nix` is the Home Manager root.
- `profiles/` contains capability-composed entrypoints for embedded and standalone Home Manager usage.
- `capabilities/` contains reusable, standalone-safe Home Manager building blocks.
- `programs/` contains common user tooling and app configuration.
- `modules/` contains feature-oriented user config like `gnome`, `hyprland`, and `_3dPrinting`.
- `assets.nix` copies tracked assets into `~/.config/userdata/`.
- `home-files/common/` contains shared raw dotfiles copied by manifest.

Private work overlay conventions:

- the main repo expects a local repo path via `my.work.repoPath`
- the private repo should export `modules/home/default.nix`
- keep the shared repo organization-neutral; use placeholders and generic examples when documenting private overlay structure
- `examples/work-overlay-mvp/` is the generic minimum-viable overlay shape for recovery or bootstrapping

Home Manager conventions:

- modules under `profiles/` and `capabilities/` should not depend on `osConfig`
- system-only mirroring from OS options should happen in the embedded Home Manager bridge in `modules/shared/home-manager.nix`
- raw dotfile copying is handled by `modules/home/capabilities/raw-files-compat.nix`
- prefer explicit manifest entries for whole app directories like `.config/nvim` instead of auto-discovering the entire home tree
- keep reserved/generated files like `.gitconfig`, `.ssh/config`, `.aws/config`, and `.config/ghostty/config` out of the raw manifest
- work-only private overlays are expected to come from a local repo path via `my.work.enable` and `my.work.repoPath`

## Feature Toggle Pattern

This repo commonly enables a feature at the OS layer and mirrors it into Home Manager.

Pattern:

1. Define an option in a system or shared module.
2. Enable it from the host config.
3. In the Home Manager module, default the matching option from `osConfig.<option>`.

Examples:

- `gnome.enable`
- `hyprland.enable`
- `_3dPrinting.enable`

This keeps host files simple and ensures user-level config follows the selected system features.

## Conventions

- Prefer reusable logic in `modules/` over putting behavior directly in `hosts/*/configuration.nix`.
- Use `default.nix` files as directory aggregators.
- Preserve cross-platform behavior where possible; many modules branch on `pkgs.stdenv.isLinux` and `pkgs.stdenv.isDarwin`.
- Pass host/user identity through `specialArgs` in `flake.nix`; existing modules rely on those values.
- Treat `modules/home/modules/hyprland/default.nix` as experimental; it is explicitly marked WIP.

## Build And Validation

Preferred commands:

- `nix flake check`
- `sudo nixos-rebuild switch --flake .#nixos-bstuart`
- `sudo darwin-rebuild switch --impure --flake "path:$PWD#bstuart-mbp-m1pro"`

Convenience aliases exist in `Makefile`:

- `make check`
- `make update`
- `make nixos-rebuild`
- `make darwin-rebuild`
- `make home-manager-personal`
- `make home-manager-work`

Notes:

- `make nixos-rebuild` uses the explicit personal host selector.
- `make darwin-rebuild` uses `path:$PWD` and `--impure` so the ignored local work repo can participate in evaluation.
- the work standalone output and work host expect a local private work module checkout at the configured path; if work is enabled and that path is missing, evaluation should fail.
- work outputs import an ignored local repo, so they should be invoked with `path:$PWD` and `--impure`.

## Editing Guidance For Future Agents

- Do not casually edit `hosts/nixos-personal/hardware-configuration.nix`; it is generated.
- Do not update `flake.lock` unless intentionally bumping inputs.
- Do not commit generated `result` symlinks or incidental `*.bkp` backup files.
- Be careful around credentials and authentication-related config:
  - `modules/shared/packages.nix` exports `CUSTOM_1P_SIGNING_PROGRAM`
  - `modules/nixos/default.nix` contains a committed `hashedPassword`; never replace this with plaintext secrets
- Preserve the existing division between system config, user config, and host-specific overrides.

## Useful Repo-Specific Facts

- NixOS setup depends on DisplayLink and the README calls out the need to prefetch its non-free blob.
- Firefox config uses NUR-provided addons in `modules/home/programs/firefox.nix`.
- Home assets are tracked in `assets/` and copied into the user's home directory declaratively.
- The repo currently supports one macOS work machine and one personal NixOS machine.
